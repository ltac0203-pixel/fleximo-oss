# Explanation: 設計原則とスコープの線引き

Fleximo が「何を作り、何を作らないか」の判断基準です。PR レビューや機能提案の際のよりどころに
してください。

## 原則

### 1. Takeout-first

MVP では**テイクアウトのみ**を扱います。イートイン・デリバリーは意図的に対象外です。

理由: 日本の小規模飲食店で「レジ待ち・注文ミスを減らす」という価値が最も大きいのはテイクアウトです。
イートインはテーブル番号管理、デリバリーは配達員連携という別ドメインを呼び込み、MVP のスコープが
爆発します。先に一点突破で品質を上げます。

### 2. Self-hostable を優先する

SaaS ビジネスのためではなく、**セルフホストできるOSS**であることを優先します。

- 決済代行以外に従量課金を持ち込まない
- 外部 SaaS に依存する機能は「差し替え可能」にする（メールは SMTP / Mailgun / SES で選べる等）
- Docker や systemd で動かせる状態を維持する

### 3. 動くより壊れないを優先

- 例外を握りつぶさない。エラーは必ずログに残す
- 本番 DB の破壊的操作（`migrate:fresh`, `db:wipe`）は禁止
- 決済は Webhook を正とする（[payment-flow.md](./payment-flow.md)）

### 4. 日本市場に固有の要件をファーストクラスで扱う

- 特定商取引法（特商法）表記ページを標準搭載
- 税込み価格表示（`price_including_tax`）
- 電話番号・郵便番号は日本フォーマットでバリデーション
- 決済は fincode（国内プロバイダ、PayPay 対応）

これらは「後で i18n する」ではなく「最初から正しく作る」べき領域です。
多言語対応は別軸で進めますが、日本の法令対応は常に優先します。

### 5. 命名とロールの一貫性

- 1つの概念に1つの名前。`UserId` / `user_id` / `uid` の混在は禁止
- ロールは3つだけ: `tenant_admin`, `tenant_staff`, `customer`
- 新しいロールを足す前に、既存ロール + 権限で表現できないか必ず検討

## やらないこと（明示）

| やらないこと                                 | 理由                                                     |
| -------------------------------------------- | -------------------------------------------------------- |
| 予約機能                                     | 飲食店予約は別プロダクトの領域（TableCheck, トレタ等）    |
| POS 機能（レジ・売上計上）                   | 外部 POS 連携の形で対応する（Square, Airレジ等を予定）   |
| 在庫・原価管理                               | MVP 外。将来的に追加可能だが急がない                     |
| デリバリー配達員マッチング                   | Uber Eats / 出前館等の領域                               |
| 独自クレジットカード情報保存                 | PCI DSS の領域になる。fincode に委譲する                 |

## 技術選定の判断基準

新しいライブラリ・サービスを追加するときは以下を確認してください。

1. **セルフホストの妨げにならないか**
2. **日本語・日本市場との相性**（UI・ドキュメント・サポート）
3. **Laravel / React の主流ベストプラクティスに沿っているか**
4. **メンテナンスされているか**（最終リリース、オープン Issue 数）
