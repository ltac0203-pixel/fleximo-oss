# Explanation: マルチテナント設計の考え方

Fleximo は1つのアプリケーションで複数の飲食店（テナント）を相乗りさせます。
どのように分離するかは複数の選択肢があり、Fleximo は **単一DB・行ベース分離**を選びました。
このドキュメントはその理由と、トレードオフを整理します。

## 選択肢と採用理由

| モデル                     | メリット                             | デメリット                                 | Fleximo |
| -------------------------- | ------------------------------------ | ------------------------------------------ | ------- |
| テナントごとに別DB         | 強い分離、バックアップ・削除が簡単   | 運用コストが高い、マイグレーションが大変   | ✗       |
| テナントごとに別スキーマ   | 一定の分離、同一インスタンスで運用可 | マイグレーション・ORM対応が面倒            | ✗       |
| **単一DB・行ベース分離**    | 実装が単純、横断検索が素直           | 誤ったクエリで漏洩リスク → 防御の工夫が必要 | **✓**   |

Fleximo のターゲットは小〜中規模の飲食店が多数相乗りするモデルで、以下を重視しました。

- **横断検索が第一級機能**（顧客は全店舗を横断して探す）。別DBだと重くなる。
- **運用は1人でも回せること**。DB数が増えると運用コストが指数的に上がる。
- **MVP で素早く試せること**。

## 漏洩リスクへの防御策

行ベース分離で最大の敵は「WHERE tenant_id を付け忘れたクエリ」です。Fleximo は三重の防御で対処します。

### 1. Eloquent Global Scope

テナント依存モデルは必ず `BelongsToTenant` トレイトを use し、Global Scope が
自動的に `WHERE tenant_id = ?` を付与します。

### 2. Policy

コントローラは `$this->authorize('view', $order)` のように Policy を通します。
Policy 内で `$order->tenant_id === $user->tenant_id` を明示的に確認します。

### 3. DB 制約

複合ユニーク制約は `(tenant_id, slug)` のように `tenant_id` を先頭に含めます。
FK も `(tenant_id, parent_id)` のペアで張り、クロステナント参照を物理的に不可能にします。

## 顧客アカウントは全テナント共通

テナント分離するのは **店舗側のリソース**（メニュー・スタッフ・注文・売上）だけです。
顧客（`customer` ロール）は単一アカウントで全店舗を利用できます。

これは UX とデータの所有権の観点からの意思決定です:

- 顧客が店舗ごとにアカウントを作るのは UX として苦痛
- 顧客の注文履歴は顧客のもの（GDPR/個人情報保護法的にも顧客所有）
- ただし**個別の注文レコード**は「顧客 × テナント」の組み合わせに属する

## Redis / キュー / ストレージ

DB 以外のリソースも tenant_id で namespace します。

- Redis キー: `tenant:{tenant_id}:...`
- キュージョブ: ペイロードに `tenant_id` を含める
- ストレージパス: `storage/app/tenants/{tenant_id}/...`

これにより、「DB はクリーンだが Redis キーが他テナントに見える」といった事故を防ぎます。

## 将来の進化

テナント数やデータ量が増えた場合、**シャーディングされた単一スキーマ**（テナント ID でDB分割）に
進化させられるよう、コードは常に `tenant_id` を引数で受け渡す形にしています。
急に別DBモデルに切り替える必要に迫られないよう、ここを崩さないでください。
